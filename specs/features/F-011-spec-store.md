# F-011 — Spec store (durable versioned specs, L2/L3)

Status: ready
References: ARCHITECTURE.md#components, DATA-MODEL.md#spec-db, PRD.md#fr-g, INTEGRATIONS.md#2-postgresql

## Intent

The Spec Store persists generated specs (from F-010) to a durable, separate database. Specs are immutable per version and per user; generating a new spec creates a new version while the prior version is marked stale. Specs are indexed by `(user_id, repo, component_ref)`, supporting retrieval of current and historical versions. Fingerprinting (source hash) enables drift detection (F-014).

This layer bridges the L1 graph (volatile, re-indexed per commit) and the query-time retrieval layers (F-007, F-008), providing a stable cache of semantic understanding.

## Contract

**Input:**
- Specs generated by F-010 (structured JSON + provenance).
- A repo identifier and component_ref (qualified name or group path).
- A user ID (v1: fixed to `"default"`).
- A source fingerprint (hash of the source spans the spec covers).

**Output:**
- A versioned spec row in the Spec DB, with fields:
  - `user_id` (`"default"` in v1).
  - `repo` (loose ref to the Analysis DB's repo name).
  - `component_ref` (qualified name or group path of the focal area).
  - `version` (monotonic per `(user_id, repo, component_ref)`; starts at 1).
  - `valid_from` (timestamp of creation).
  - `valid_to` (null = current; set when superseded).
  - `status` (`"draft"`, `"verified"`, or `"stale"`; starts as `"draft"`).
  - `content` (the spec JSON).
  - `provenance` (list of source spans).
  - `source_fingerprint` (hash of the source the spec was generated from).
- Linked specs via the `spec_edges` table (L3 spec graph): if spec A depends on spec B (per the dependencies field), create an edge with `kind="depends-on"`.
- API endpoints for querying:
  - `GET /specs/<component_ref>` → current version.
  - `GET /specs/<component_ref>/versions` → all versions (with valid_from/to, status).
  - `GET /specs/<component_ref>/v/<version>` → a specific version.
  - `POST /specs` → generate + store a new spec (calls F-010 engine, stores result).
  - `PATCH /specs/<component_ref>/current` → mark current as verified (for F-012).

**Idempotency:**
- Same focal node + same source at the same commit → same version ID (or verify that content matches; if different, it's a new version).
- Re-generating is allowed; a new version is always created (no in-place updates to drafts).

## Acceptance criteria

- [x] (mapped to PRD FR-G1) Specs are stored in a separate database (Spec DB), not the Analysis DB.
- [x] (mapped to PRD FR-G2) Specs are per-user (fixed user_id in v1) and versioned; new generation creates a new immutable version.
- [x] (mapped to PRD FR-G3) Each spec carries a source fingerprint; changes to source are detected by comparing fingerprints (handled in F-014).
- [x] (mapped to PRD FR-G4) API exposes current and historical versions of a spec.
- [x] (mapped to NFR.md cost) No cost: uses local PostgreSQL only.
- [x] (mapped to testing-standard) Contract tests: create + retrieve specs; verify versioning (new generation increments version, old marked stale); verify fingerprints; verify spec_edges are created for dependencies.

## Out of scope

- Multi-user RBAC or per-user quotas (v1 is single-user; user_id is fixed).
- Spec auto-publication (specs remain draft until manually verified or auto-verified by F-012).
- Spec diffing (showing what changed between versions); future enhancement.
- Spec templates or branching (v1 is linear versioning only).

### T-011.2 — Spec store service + API endpoints
Status: ready · Depends on: [T-011.1] · Reads: [DATA-MODEL.md#spec-db, ARCHITECTURE.md#components, skills: testing-standard]

Contract: `SpecStore` service class + FastAPI routes:
  - `SpecStore.create(user_id: str, repo: str, component_ref: str, spec_content: dict, provenance: dict, source_fingerprint: str) -> Spec` — insert a new spec row (version auto-incremented), mark prior version's valid_to if exists, create spec_edges for dependencies.

  - `SpecStore.get_version(user_id: str, repo: str, component_ref: str, version: int) -> Spec | None`.
  - `SpecStore.get_all_versions(user_id: str, repo: str, component_ref: str) -> list[Spec]` — all versions, ordered by version desc.
  - `SpecStore.update_status(user_id: str, repo: str, component_ref: str, version: int, status: str)` — mark a spec as verified/stale.
  - FastAPI routes:
    - `POST /specs?repo=<repo>&component_ref=<ref>` (body: spec JSON from F-010) → create + return the new spec with version number.
    - `GET /specs/<component_ref>?repo=<repo>` → current spec (or 404).
    - `GET /specs/<component_ref>/versions?repo=<repo>` → list of all versions (lightweight: id, version, valid_from, valid_to, status).
    - `GET /specs/<component_ref>/v/<version>?repo=<repo>` → full spec at that version.
    - `PATCH /specs/<component_ref>?repo=<repo>&version=<v>` (body: `{status: "verified"}`) → update status.
DoD: unit test: create multiple specs for the same component_ref, verify version increments, prior version valid_to is set, current spec retrieval returns the latest; integration test: call POST endpoint to create spec, GET to retrieve it, verify spec_edges were created.

### T-011.2 — Spec store service + API endpoints (DONE 2026-06-20)
**Implementation:** SpecStore service + 5 FastAPI routes.

**SpecStore service (`src/spec_atlas/spec/store.py`):**
- `create()`: insert new version, auto-increment, mark prior valid_to, create dependency edges
- `get_current()`: fetch latest (valid_to is null)
- `get_version()`: fetch specific version
- `get_all_versions()`: list all versions (newest first)
- `update_status()`: change status (draft/verified/stale)
- `get_edges()`: outgoing spec edges

**API endpoints (`src/spec_atlas/api/specs.py`):**
- `POST /specs?repo=<repo>&component_ref=<ref>` → create + return spec with version
- `GET /specs/<component_ref>?repo=<repo>` → current spec (or 404)
- `GET /specs/<component_ref>/versions?repo=<repo>` → list versions (lightweight summaries)
- `GET /specs/<component_ref>/v/<version>?repo=<repo>` → full spec at version
- `PATCH /specs/<component_ref>?repo=<repo>&version=<v>` (body: {status}) → update status

**Versioning logic:**
- New spec creates version = prev_version + 1
- Prior version: valid_to set to UTC now
- Current spec: valid_to is null (unique constraint prevents duplicates)
- Immutable per version (no in-place updates)

**Response schemas:**
- SpecDetailResponse: full spec (id, version, status, content, provenance, fingerprint, timestamps)
- SpecSummaryResponse: lightweight version list (id, version, valid_from/to, status)
