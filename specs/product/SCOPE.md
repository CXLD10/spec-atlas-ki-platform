## In scope (v1 / Phase 0–4)

### Foundation (Phase 0)
- Ingest code repos (local path / public git URL) in **multiple languages** via tree-sitter (initial packs: Python + TypeScript/JS).
- L1 code knowledge graph: symbols + edges (imports, calls, inherits, defines) with confidence.
- API_CONTRACT.md: canonical REST API + field mappings (backend ↔ frontend).
- Frontend wiring: routes, ask box, answer view, citation rendering.

### Knowledge Synthesis (Phases 0–2)
- L2 specs generated by an LLM on demand (the **Specify tool**: select a node, click "generate", store versioned spec).
- L3 spec graph: links from real code edges + multi-source cross-references.
- L4 `group.md` tree: hierarchical summaries with file:line receipts.
- Embeddings for group summaries + specs; **hierarchical retrieval** (vector over groups → descend).

### Retrieval & Answer (Phases 0–1, enhanced in 3)
- **Ask-anything** with dual-path routing (detail → L1 graph; big-picture → spec/group path).
- Multi-source citations: `{file, line}` for code; `{page, bbox}` for PDF; `{section}` for Markdown.
- **Conversation memory** (Phase 3): facts persist across sessions; retrieved by relevance; accessible via API + MCP.

### Multi-Source (Phases 1–3)
- **SourceUnit abstraction**: ingest code, PDF, Markdown, Excel (Jira/git history optional).
- **PDF adapter** (PyMuPDF): text + bbox preservation for precise citations.
- Embeddings normalized across source types.

### Storage & Agents (Phases 0–3)
- Spec store (separate DB, per-user, versioned, fingerprinted).
- MCP server: `search`, `get_spec`, `get_group`, `get_memory_facts`.
- HTTP API (mirrors MCP for non-MCP callers).
- Living onboarding docs: `group.md` tree browsable in UI.

## Out of scope (Phase 0–4 sprint)
- **Spec verification**: verifying invariants against code (saved for v1).
- **Drift detection**: marking stale specs when code changes (saved for v1).
- **Eval harness**: head-to-head comparison vs. raw-node baseline (saved for v1).
- Jira/git history adapters (optional/stretch in Phase 3).
- Multi-user / org RBAC / SaaS deployment (single-user first).
- Auto-applying code changes (Spec-Atlas informs agents; agents act).
- Private-repo credential storage (public repos only).
- Perfect cross-language call resolution (best-effort + confidence scores; LLM spec layer carries semantic load).
- Runtime/dynamic analysis or any **execution of repo code** (static only; pipeline reads files + calls model, nothing else).

## Non-goals
Not a chatbot; not generic doc-RAG. Answers are graph/spec-grounded with mandatory provenance, scoped to the indexed repo. Not a paid service (any non-free dependency needs an ADR).

## MVP boundary
Index a multi-language repo → graph + specs + `group.md` tree (= onboarding docs) → one grounded, routed, cited answer → generate + store one spec → an agent reads it via MCP. Everything else layers on after that vertical slice works.
