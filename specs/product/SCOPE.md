# SCOPE.md — Spec-Atlas

Status: ready
> Focused sub-spec. Full requirements in `PRD.md` §8.

## In scope (v1)
- Ingest a repo (local path / public git URL) in **multiple languages** via tree-sitter (initial packs: Python + TypeScript/JS).
- L1 code knowledge graph: symbols + edges (imports, calls, inherits, defines) with confidence.
- L2 specs generated by an LLM (the Specify tool); L3 spec graph (links from real code edges); L4 `group.md` tree of summaries.
- Embeddings for group summaries + specs; **hierarchical retrieval** (vector over groups → descend).
- **Dual-path retrieval with a router**: detail/symbol questions → code graph; big-picture questions → spec/group path.
- Ask-anything with `file:line` provenance; spec store (separate DB, per-user, versioned, fingerprinted); spec verification; drift detection; MCP server; living onboarding docs; a head-to-head eval harness.

## Out of scope (v1)
- Runtime/dynamic analysis or any **execution of repo code** (static only; the pipeline reads files + calls a model, nothing else).
- Auto-applying code changes (Spec-Atlas informs agents; agents act).
- Multi-tenant SaaS, billing, org RBAC (single-user first).
- Private-repo credential storage (public repos only in v1).
- Perfect cross-language call resolution (best-effort + confidence scores; the LLM spec layer carries the semantic load).

## Non-goals
Not a chatbot; not generic doc-RAG. Answers are graph/spec-grounded with mandatory provenance, scoped to the indexed repo. Not a paid service (any non-free dependency needs an ADR).

## MVP boundary
Index a multi-language repo → graph + specs + `group.md` tree (= onboarding docs) → one grounded, routed, cited answer → generate + store one spec → an agent reads it via MCP. Everything else layers on after that vertical slice works.
